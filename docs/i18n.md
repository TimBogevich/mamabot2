# Internationalization (i18n)

Last updated: 2026-06-17

## Overview

The i18n module (at `functions/src/i18n.js`) provides Russian and English language support for the MamaBot Telegram bot. Locale files are stored in `functions/src/locales/` and contain all user-facing strings. Each file represents one language and has an identical key structure.

A duplicate copy is maintained at `functions/src/i18n/` (for key parity tests).

### Files

| File | Language |
|------|----------|
| `functions/src/locales/ru.json` | Russian |
| `functions/src/locales/en.json` | English |

### Key Structure

Keys use dot-notation with segments in snake_case. Nested objects are resolved via dot, for example:

```
onboarding.welcome         → "👋 Добро пожаловать..."
error.generic              → "❌ Произошла ошибка..."
```

### Placeholders

Strings may contain placeholders in curly braces:

| Placeholder | Description | Example |
|-------------|-------------|---------|
| `{week}` | Pregnancy week number | `"{week} неделя беременности"` |
| `{date}` | Date in YYYY-MM-DD format | `"Дата сохранена: {date}"` |
| `{code}` | Partner invitation code | `"Твой код: {code}"` |
| `{size}` | Fetus size (comparison) | `"📏 Size: {size}"` |
| `{weight}` | Fetus weight in grams | `"⚖️ Weight: {weight} g"` |

### Key Naming Convention

1. **Keys in snake_case** — e.g., `baby_development`, `meal_types`, `vitamin_recommendations`.
2. **Top-level categories** — singular nouns: `menu`, `mood`, `nutrition`.
3. **Nested objects** — for grouping related keys: `meal_types.breakfast`, `meal_types.lunch`.
4. **Consistency** — identical keys in both language files; every key exists in both `ru.json` and `en.json`.

### How to Add a New Key

1. Add the key to `ru.json` with the Russian translation.
2. Add the **exact same key** to `en.json` with the English translation.
3. If the key belongs to a new category, create a new top-level object.
4. Run tests to check key parity:
   ```bash
   cd functions
   npx vitest run src/i18n/
   ```
5. Tests verify:
   - Both files exist and contain valid JSON
   - All keys (including nested) are identical in both files
   - No empty strings
   - `ru.json` contains Cyrillic characters

## API

### `t(userId, key, vars?)`

Asynchronously resolves a dot-notation key to a localized string for the specified user.

#### Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `userId` | `number \| string` | Telegram chat ID (used as document ID in Firestore) |
| `key` | `string` | Dot-notation key, e.g. `'menu.week_info'` |
| `vars` | `Record<string, string \| number>` | Optional substitutions for `{{placeholder}}` |

#### Returns

`Promise<string>` — the resolved localized string.

#### Error Behavior

- Returns the raw `key` if the key is missing from both locale files (no error thrown)
- Returns `String(key)` for non-string keys (e.g., `t(chatId, 123)` → `'123'`)

#### Chain of Fallbacks

1. User's language (from Firestore `users/{userId}.language`)
2. Russian (`'ru'`) if the user has no language preference
3. Raw `key` if the key is absent from both locale files

#### Examples

```js
// Simple key resolution
const msg = await t(chatId, 'menu.week_info');
// → "📋 Информация о неделе" (ru) or "📋 Week Info" (en)

// With variable substitution
const msg = await t(chatId, 'onboarding.welcome', { name: 'Анна' });
// → "👋 Добро пожаловать в MamaBot, Анна!..."

// Multiple substitutions
const msg = await t(chatId, 'onboarding.week_calculated', {
  week: '5',
  edc: '2026-12-01',
});
// → "🤰 Ты на 5 неделе беременности! Предполагаемая дата родов: 2026-12-01."

// Fallback for missing key
const msg = await t(chatId, 'nonexistent.key');
// → 'nonexistent.key' (no error)

// Non-string key
const msg = await t(chatId, 123);
// → '123'
```

### `setLanguage(userId, lang)`

Sets a user's language preference in Firestore.

#### Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `userId` | `number \| string` | Telegram chat ID |
| `lang` | `'ru' \| 'en'` | Language code |

#### Returns

`Promise<string>` — the set language code.

#### Error Behavior

Throws `Error` if `lang` is not `'ru'` or `'en'`.

#### Examples

```js
// Set English
await setLanguage(chatId, 'en');

// Set Russian
await setLanguage(chatId, 'ru');

// Error on invalid code
await setLanguage(chatId, 'fr'); // throws Error
```

├── already_registered   — message for repeat /start
├── ask_lmp              — prompt for LMP date (format DD.MM.YYYY)

├── date_in_future       — future date error

├── week_calculated      — confirmation with {{week}} and {{edc}}
├── edd_confirm           — auto-EDD message with Верно/Исправить buttons (with {{edd}})
├── edd_correct           — button text «✅ Верно» (callback: onboarding_confirm_edd)

> **Date format note:** Starting from task FN-025, keys `ask_lmp`, `lmp_format_hint`, and `invalid_date` use the **DD.MM.YYYY** format (previously ГГГГ-ММ-ДД). Users enter dates with dots, e.g., `15.03.2026`. Internally the date is stored in ISO format YYYY-MM-DD.

## Variable Substitution

The `{{variable}}` template in a string is replaced with the corresponding value from the `vars` object. Undefined variables are left untouched (no error thrown).

```js
// Key: "welcome": "Привет, {{name}}!"
await t(userId, 'welcome', { name: 'Мария' });
// → "Привет, Мария!"

// Key: "week_calculated": "Ты на {{week}} неделе!"
await t(userId, 'week_calculated', {});  // empty vars
// → "Ты на {{week}} неделе!" (placeholder remains)
```

## Usage

Files are loaded via `require()` in `i18n.js`:

```js
const ru = require('./locales/ru.json');
const en = require('./locales/en.json');
```

The `t(chatId, key, params)` function is available via:

```js
const { t } = require('./i18n');
const text = await t(chatId, 'onboarding.welcome', { name: 'Анна' });
```

## Firestore Integration with setLanguage

The module uses Firestore to persist user language preferences:

- **Read:** `getUser(userId)` — retrieves the document from `users/{chatId}`
- **Write:** `updateUser(userId, { language: lang })` — updates the `language` field

The `language` field in Firestore has type `'ru' | 'en'`.
